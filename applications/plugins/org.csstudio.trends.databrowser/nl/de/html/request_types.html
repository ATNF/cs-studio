Data Request Types

Die Config View erlaubt die Auswahl von "Request Types" für archivierte Daten.

Man beachte, dass diese Optionen nur die Abfrage von historischen Archivdaten betreffen. Sie haben keinen Einfluss darauf, wie Daten ins Archiv geschrieben werden. Einige PVs aktualisieren den Messwert mit 10Hz und schreiben einmal pro Minute ins Archiv. Andere aktualisieren den Messwert nur, wenn eine Änderung vorliegt und archivieren jede Änderung. Wie ein PV Daten aktualisiert, wird durch das Front-End-Gerät bestimmt, welches den PV unterstützt. Die Archivierung hängt von der Archiv-Sampling-Engine ab. Der Datenbrowser ist nur für die Darstellung von 'live'- und Archiv-Daten zuständig.

Optimized Request

Dies ist die Voreinstellung. Der optimierte Abfrage-Typ schaltet automatisch zwischen den Originawerten und der Min/Max/Mittelwert-Information hin und her. Unter CSS/Preferences/../Trends/DataBrowser kann man die Anzahl der für die Datenabfrage benutzten 'bins' konfigurieren. Beispiel: Man konfiguriert 500 'bins' und der Graph habe ein Zeitfenster von einem Tag. Wenn weniger als 500 Daten im Archiv sind, erhält man 500 Original-Daten. Wenn es mehr als 500 sind, erhält man für jeden 'Bin die Mittelwert-Daten kombiniert mit den Minima und Maxima.

Wenn man "hineinzoomt", holt der Datenbrowser Werte für das neue Zeitfenster, die mehr Details zeigen können und schließlich auf Originaldaten umschalten können.

Der optimierte Abfrage-Typ versucht hier, die Datenmenge, die über das Netz übertragen und im Plot behandelt wird, zu minimieren, während gleichzeitig darauf geachetet wird, dass genug Information über das Verhalten des PV vorhanden ist.

Raw Request

Selten lohnt es sich, Einschränkungen im optimierten Abfrage-Mechanismus durch Erzwingen einer Rohdatennahme zu überschreiben. Aber meist wird die Anforderung von Rohdaten nichts bringen, nur wird der Computer Speicherprobleme haben.

Wenn man zur Nachbearbeitung Rohdaten für ein großes Zeitfenster benötigt , sollte man den Export View benutzen.
